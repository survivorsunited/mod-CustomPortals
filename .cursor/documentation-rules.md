# Documentation Rules - Custom Portals Mod

## General Principles
- Maintain consistency across all documentation files
- Use proper Minecraft terminology
- Document portal mechanics clearly
- Keep configuration options well-documented
- Update documentation when adding new features

## Portal System Documentation
- Document portal creation process
- Explain portal linking mechanics
- Describe rune effects and usage
- Include configuration options and defaults
- Provide troubleshooting guidance

## File Naming Conventions
- Use kebab-case for documentation files
- Follow Minecraft naming conventions
- Use descriptive names for feature documentation

## Files to Update Together
When making changes, update:
1. `README.md` - Project overview
2. `BUILD.md` - Build instructions (if applicable)
3. Configuration documentation
4. Any feature-specific documentation

## Image Links (if applicable)
- Use minecraft.wiki image sprites when referencing vanilla blocks/items
- Format: `https://minecraft.wiki/images/[Type]Sprite_[item-name].png?[hash]`
- Types: ItemSprite, BlockSprite, Invicon
- Always include alt text and titles

## Wiki Links (if applicable)
- Link to official Minecraft wiki pages for vanilla content
- Format: `https://minecraft.wiki/w/[Item_Name]`
- Use proper page names (spaces become underscores)
- Include title attributes for accessibility

## Markdown Best Practices
- Use proper heading hierarchy
- Include code blocks for configuration examples
- Keep line lengths reasonable for readability
- Use clear, concise language
- Include examples where helpful

## Configuration Documentation
- Document all configuration options
- Explain default values
- Provide usage examples
- Note any important behavior changes
